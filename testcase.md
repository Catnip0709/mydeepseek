**MyDeepSeek 测试用例**

**前置条件**

- 本项目使用 ES Modules（`<script type="module">`），**必须通过 HTTP 服务器访问**，不支持 `file://` 协议直接打开
- 推荐启动方式：`python3 -m http.server 8080`，然后访问 `http://localhost:8080`
- 默认浏览器需开启 localStorage，且允许访问外部 CDN / DeepSeek API（仅真实 API 冒烟场景需要）

**测试范围**

- 覆盖端：桌面端、手机端
- 覆盖模式：夜间模式、日间模式
- 覆盖类型：静态结构、模块化架构、UI/响应式、核心交互、异常分支、性能优化、真实 API 冒烟

**优先级说明**

- `P0`：阻塞上线 / 核心主链路，回归必须执行
- `P1`：高频功能 / 关键兼容性，常规回归建议执行
- `P2`：低频功能 / 边界与增强项，版本回归或专项时执行

**执行建议**

- 每次改动至少执行：静态可用性、API Key、会话管理、消息发送主链路、指令管理、角色卡管理、真实 API 冒烟
- 涉及 UI / CSS 改动时，补测：主题与视觉一致性、响应式与布局、日间模式兼容
- 涉及数据结构 / 存储改动时，补测：存储用量显示与保护、数据损坏自动修复、数据兼容与边界
- 涉及消息流 / LLM 调用改动时，补测：停止生成、重新生成、上下文与 token 提示、请求超时控制、单角色聊天、群聊消息编排

**十一A、记忆策略、摘要与滑动窗口**

- 新用户或本地未保存 `dsMemoryStrategy` 时，默认记忆策略为 `full`（全量发送）
- 默认全量模式下，单聊发送 payload 使用全量历史（仍受上下文裁剪），不依赖摘要
- 默认全量模式下，不自动生成摘要、不自动迁移旧摘要；群聊也不注入 `groupContext.summary`
- 切换到滑动窗口策略后，当消息总数达到摘要阈值，会自动生成摘要，但最近 `20` 条消息仍以原文参与上下文
- 切换到滑动窗口策略后，摘要更新后 `summaryCoversUpTo` 不会推进到当前末尾，而是保留最近 `20` 条原文窗口
- 切换到滑动窗口策略后，最近 `20` 条消息删除、编辑后，发送给模型的上下文仍符合滑动窗口规则，不会把最近窗口整体吞进摘要
- 切换到滑动窗口策略后，老会话若携带旧摘要（无当前 `summaryVersion`），在迁移完成前不会把旧摘要作为可用记忆注入 prompt
- 切换到滑动窗口策略后，页面初始化会后台扫描并迁移旧摘要会话，当前活动会话优先，其余旧会话继续顺序迁移
- 切换到滑动窗口策略后，无 API Key 启动再补填 API Key，会触发旧摘要迁移，不需要手动刷新页面
- 切换到滑动窗口策略后，切换到仍未迁移的旧 tab 时，会按需触发该 tab 的旧摘要迁移
- 摘要生成、更新、重建和快照对比时，会过滤"预览网页"产生的 HTML 请求与 HTML 代码消息

- 群聊中若存在旧摘要，迁移前不应继续把旧摘要注入 `groupContext.summary`
- 新生成、更新、重建后的摘要都会写入当前 `summaryVersion`，刷新页面后版本标记不丢失
- 摘要生成和更新使用 `state.selectedModel` 指定的模型

**二十、边界与异常流程**

- 页面在没有任何会话内容时能正常工作
- localStorage 中旧数据结构仍可兼容读取
- 老会话数据中存在旧 `memoryLimit` 字段时不影响当前逻辑
- 网络异常时，页面不会整体卡死
- API 返回错误时，会显示错误信息
- 失效 Key、网络错误、后台切回等异常路径下，不应破坏已有会话内容
- 刷新页面后，会话、设置、指令、主题、收藏、模型选择等本地数据可恢复
- 多次快速切换设置项，不应导致明显状态错乱
- 单聊请求失败（API 错误、网络错误、超时、手动停止、后台中断）后，该失败轮次不会被写入摘要
- 群聊请求失败（路由失败、角色生成失败、手动停止、超时）后，该失败轮次不会触发摘要更新
- 失败轮次之后恢复正常发送，后续摘要内容不应包含失败提示文案或只存在用户提问、没有 AI 回复的残缺轮次

**二十一、存储用量显示与保护**

- 侧边栏底部显示"本地存储容量 xxx/5MB(x%)"文案
- 存储用量在发送消息、保存指令、保存 API Key、收藏消息后自动更新
- 存储用量 < 95% 时显示灰色文字，无感叹号图标
- 存储用量 ≥ 95% 时文字变红，出现红色感叹号图标

**三十二、群聊消息发送与编排**

- 角色回复内容符合其性格设定（傲娇角色说"哼"等）
- 多个角色回复时，依次显示（先角色A，再角色B）
- 普通群聊场景下，优先由 `2-3` 个与当前话题相关的角色参与；明显私聊/点名场景可只让 `1` 个角色回复
- Agent 群聊使用稳妥版工具集，仅暴露 `character_reply`、`narrate`、`search_conversation`、`query_story_archive`
- 每个角色回复最多 `6` 句话
- Agent 群聊中，角色台词与旁白都应通过工具调用产出，不应出现 assistant 直接输出整段可见正文
- Agent 群聊中，查询工具只在当前注入上下文不足时才调用，不应每轮都先检索再说话
- 未被选中的角色如果与已回复角色或当前话题强相关，会自动追问
- 追问判断合理：不相关时不追问，相关时可自然接话，不会把"不是"之类否定误判成要插话
- 旁白可在场景切换、动作衔接、多人沉默/对视、气氛变化时简短插入，但不连续刷屏
- 群聊编排读取最近历史时，会过滤"预览网页"产生的 HTML 请求与 HTML 代码消息，不把网页代码当作角色上下文
- Agent 群聊若尚未产出任何角色/旁白消息，可回退到传统编排兜底
- Agent 群聊一旦已经产出过角色/旁白消息，后续异常时不应整段清空已显示内容，不应出现明显"撤回感"
- 群聊中途发生普通异常（非 AbortError）时，已完成并进入预览队列的角色/旁白消息会先落盘保留，再显示错误提示
- Agent 群聊默认最低参与人数为 `2` 个不同角色；在明确私聊、独处、点名单人或其他角色明显不在场时，才允许降为 `1`
- Agent 群聊达到最低参与人数后，若仍存在明显自然接话者，可继续自然扩张到 `3-4` 人；不应第一个角色说完就立刻结束
- Agent 群聊若首轮只生成了 `1` 个角色，且不属于单人例外场景，应自动补至少 `1` 个相关角色发言
- Agent 群聊中，引用回复只决定“被引用角色先说”，不应自动把整轮群聊降成单人回复；若场景仍是多人在场，其他强相关角色仍可继续加入
- 普通群聊里即使正文只点到一个角色名，只要没有明确独处/私聊/其他人不在场，也不应自动退化为单人场景
- 发送中点击"停止"可中断群聊生成
- 中断后已生成的角色消息保留，未生成的角色不显示
- 中断的角色消息标记为"生成中断"
- 群聊发送失败时显示错误提示（toast）
- 无 API Key 时在群聊中发送会弹出 Key 输入面板
- 群聊首轮对话后可自动生成会话标题
- 群聊消息支持 Markdown 渲染
- 长时间群聊后，历史对话正确传递给编排器

- 追问判断被中断时，已完成的保留
- Agent 群聊已显示 A/B/旁白 后，再发生普通异常，不会因为最终 `renderChat()` 导致这些已完成消息消失
- Agent 群聊已显示 A/B/旁白 后，再发生普通异常，允许保留已完成内容并同时显示失败 toast
- Agent 群聊完全零产出时发生异常，允许回退到传统编排；若传统编排也失败，则不应伪造不存在的角色消息

**三十八、群聊多轮互动**

- 用户发消息后，角色A回答 → 角色B回应 → 角色A再回应（多轮互动）
- 多轮互动最多 `30` 轮，不会无限循环
- 每轮最多一个角色回应
- 每个角色在整次编排中最多说 `3` 次
- 多轮互动中允许穿插 `0-1` 条简短旁白串联气氛，但不应连续刷屏
- 任何一轮没人想说话时自动结束
- 多轮互动中角色能看到本轮其他角色已说的话（不重复）
- 多轮互动中停止按钮可以中断

**三十九、群聊导出**

- 群聊对话导出为 TXT 时，角色消息显示正确的角色名（不是"DeepSeek"）
- 用户消息显示为"我"
- 单聊消息导出不受影响

**三十九B、单角色聊天**

- 角色卡列表中每个角色有"单独聊天"按钮（聊天图标）
- 点击后创建新会话，tab 类型为 single-character，存储 characterId
- 点击后自动关闭角色卡管理面板和侧边栏
- tab 标题默认显示角色名（无自定义标题时）
- 单角色聊天不自动生成标题，tab 名称始终显示角色名（不会被动态生成的标题覆盖）
- 用户可手动重命名，重命名后显示自定义标题（覆盖角色名）
- tab 前显示 🎭 图标（与群聊 👥、单聊无图标区分）
- 聊天区顶部显示"正在与角色对话"+ 角色名彩色标签
- 发送消息时自动注入角色 system prompt（性格、背景、外貌、说话风格、口头禅）
- system prompt 通过 buildPayloadMessages 在 payload 最前面插入，不影响本地消息存储
- AI 回复以角色身份和风格回复（不是默认 DeepSeek 助手风格）
- AI 回复消息上方显示角色名标签（带角色颜色），左侧有彩色边框
- 流式生成过程中也显示角色名标签和彩色边框
- 流式完成后全量渲染（finalizeMessage）时角色名标签不丢失
- 支持深度思考模式
- 支持编辑用户消息和重新生成
- 支持停止生成
- 支持版本历史切换
- 支持导出为 TXT（AI 回复显示角色名，不是"DeepSeek"）
- 删除角色时，如果该角色有单独聊天会话，弹出提示阻止删除（引用类型显示为"角色对话"）
- 删除角色后，已有对话中角色 system prompt 丢失（getCharacterById 返回 null），回退为普通对话
- 切换到其他 tab 再切回，消息正常显示（DOM 缓存正常）
- 单角色聊天的消息格式与单聊一致（用户蓝色气泡，AI 灰色气泡），不使用群聊的角色消息样式
- 单角色聊天支持 token 统计和上下文上限提醒（与单聊一致）
- 旧版数据（无 single-character 类型）不受影响

**四十一、数据兼容与边界**

- 旧版数据（tab 无 type 字段）默认作为单聊处理
- 群聊中旧数据（role 为 assistant 的消息）正确渲染为角色消息样式
- 删除角色前，如果该角色被对话或群聊引用，弹出提示要求先删除相关对话，不允许直接删除角色
- 删除未被引用的角色后，引用该角色的群聊不会崩溃（getCharacterById 返回 null 被 filter 过滤）
- 删除角色后，群聊历史消息中的角色名仍能正确显示（因为保存在消息数据中）
- localStorage 容量监控包含角色卡数据（dsCharacters）和收藏数据（dsFavorites）
- dsCharacters 数据损坏时自动重置为空数组
- dsFavorites 数据损坏时自动重置为空数组
- 群聊名称为空时默认显示"群聊"
- 新建群聊面板每次打开时清空之前的选择
- 新建群聊面板每次打开时清空之前输入的群聊名称
- 旧版消息数据中无 `id` 字段时，`initializeData` 会自动补生成唯一 ID
- `dsSelectedModel` 中存储了非法值（如空字符串、不存在的模型名）时，回退到默认值 V4-Flash
- `dsDeepThink` 中存储了非法值时，回退到 `false`（关闭深度思考）

**四十二、背景信息功能**

- "背景"入口在所有会话类型（单聊、群聊、角色对话）中均可用
- 点击菜单中的"背景"可打开背景信息面板，且菜单自动关闭
- 背景信息面板包含"你的角色名称"输入框和"对话背景"文本域

**四十二B、背景信息注入 LLM**

- 单聊中设置了背景信息后，发送消息时 LLM 回复会体现背景信息（如称呼用户设定的角色名）
- 单聊中同时设置了角色名称和对话背景，两者都会注入到 system prompt
- 单角色对话中设置了背景信息，角色回复会融入背景设定
- 群聊中设置了背景信息，角色回复会融入背景设定（通过 groupContext 传递）
- 未设置背景信息时，对话行为与之前完全一致（不注入额外 system prompt）
- 背景信息作为 system prompt 注入，不受历史消息裁剪影响（始终存在）
- 背景信息中包含特殊字符（换行、引号等）时，不会导致 prompt 构建错误

**四十三B、预览网页模式与 HTML 历史隔离**

- 开启"预览网页"后发送消息，AI 侧先显示"正在生成网页代码"进度态，完成后一次性落盘为 HTML 代码块
- 在预览网页模式下发送第二条消息、重新生成消息、编辑用户输入后重新生成，仍走网页生成进度态，而不是普通流式聊天样式
- 预览网页模式构建剧情上下文时，会过滤历史 HTML 网页代码，只保留摘要与最近的非 HTML 消息
- 当记忆策略为全量且当前没有可用摘要时，预览网页模式会临时生成一次性摘要辅助理解剧情
- 正常聊天、摘要生成、剧情档案馆、群聊最近历史都会过滤 HTML 网页代码，不让网页代码污染后续文本创作
