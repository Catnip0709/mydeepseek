**MyDeepSeek 测试用例**

**前置条件**

- 本项目使用 ES Modules（`<script type="module">`），**必须通过 HTTP 服务器访问**，不支持 `file://` 协议直接打开
- 推荐启动方式：`python3 -m http.server 8080`，然后访问 `http://localhost:8080`
- 默认浏览器需开启 localStorage，且允许访问外部 CDN / DeepSeek API（仅真实 API 冒烟场景需要）

**测试范围**

- 覆盖端：桌面端、手机端
- 覆盖模式：夜间模式、日间模式
- 覆盖类型：静态结构、模块化架构、UI/响应式、核心交互、异常分支、性能优化、真实 API 冒烟

**优先级说明**

- `P0`：阻塞上线 / 核心主链路，回归必须执行
- `P1`：高频功能 / 关键兼容性，常规回归建议执行
- `P2`：低频功能 / 边界与增强项，版本回归或专项时执行

**执行建议**

- 每次改动至少执行：静态可用性、API Key、会话管理、消息发送主链路、指令管理、角色卡管理、真实 API 冒烟
- 涉及 UI / CSS 改动时，补测：主题与视觉一致性、响应式与布局、日间模式兼容
- 涉及数据结构 / 存储改动时，补测：存储用量显示与保护、数据损坏自动修复、数据兼容与边界
- 涉及消息流 / LLM 调用改动时，补测：停止生成、重新生成、上下文与 token 提示、请求超时控制、单角色聊天、群聊消息编排

**十一、上下文与 token 提示**

- 用户消息下方显示"本次正文 x 字，约 y tokens；历史记忆约 z tokens；本轮输入共约 a tokens"
- 用户消息 token 提示在日间/夜间模式都清晰可见
- 删除历史消息后，旧用户消息底部 token 提示会实时刷新
- 上下文上限提醒基于真实发送给模型的裁剪后消息计算
- 当上下文接近上限时，会出现提醒文案
- 上下文上限提醒中的"总结指令"文案与当前摘要策略一致，提示生成约 `6000` 字总结
- 提醒中的"总结指令"可复制，复制后图标变为绿色对勾

**十一A、记忆摘要与滑动窗口**

- 当消息总数达到摘要阈值后，会自动生成摘要，但最近 `20` 条消息仍以原文参与上下文
- 摘要更新后，`summaryCoversUpTo` 不会推进到当前末尾，而是保留最近 `20` 条原文窗口
- 最近 `20` 条消息删除、编辑后，发送给模型的上下文仍符合滑动窗口规则，不会把最近窗口整体吞进摘要
- 老会话若携带旧摘要（无当前 `summaryVersion`），在迁移完成前不会把旧摘要作为可用记忆注入 prompt
- 页面初始化后会后台扫描并迁移旧摘要会话，当前活动会话优先，其余旧会话会继续顺序迁移
- 无 API Key 启动后，补填 API Key 会触发旧摘要迁移，不需要手动刷新页面
- 切换到仍未迁移的旧 tab 时，会按需触发该 tab 的旧摘要迁移
- 单聊中完成 `60章 -> 生成番外 -> 删除番外 -> 写第61章` 链路后，最近章节不应明显失忆，token 不应从高位断崖式掉到仅剩摘要级别
- 群聊中若存在旧摘要，迁移前不应继续把旧摘要注入 `groupContext.summary`
- 新生成、更新、重建后的摘要都会写入当前 `summaryVersion`，刷新页面后版本标记不丢失

**十二A、TXT 文件上传**

- 仅支持上传单个 `.txt` 文件
- 上传非 `.txt` 文件时提示"仅支持上传 .txt 文件"
- 上传空文件或纯空白文件时提示"文件内容为空，无法发送"
- 上传后输入区上方显示文件状态条，包含文件名、字数和处理模式
- 输入框为空时，主按钮显示为 `+`，点击后弹出包含"背景"和"TXT 上传"的菜单
- 输入框有内容时，主按钮从 `+` 切换为 `发送`；发送中再切换为 `停止`
- 输入框为空时点击 `+` 不会直接发送消息，只会打开菜单
- `+` 菜单点击页面其他区域后会自动关闭
- 点击 `+` 菜单中的"背景"或"TXT 上传"后，菜单会先关闭再执行对应操作
- 输入框从空变为非空时，若 `+` 菜单处于打开状态，会自动关闭并切换为 `发送`
- 点击状态条"移除"可清除待发送文件
- 字数 `<= 5000` 时，状态条显示"全文发送"
- 字数在 `5000-25000` 时，状态条显示"摘要发送"，发送时先自动总结再发给 DS
- 字数在 `5000-25000` 时，点击发送后，摘要进行中主按钮切换为红色 `停止` 样式，提示系统正在工作
- 字数 `> 25000` 时，状态条显示超限，发送时提示"文件字数超出上限（25000字），请裁剪后再试"
- 已上传 txt 但用户未输入问题时，发送会提示"请输入你的问题后再发送文件"
- 第二档发送时，状态条先显示"正在总结..."，再显示"正在发送给 DS..."
- 第二档摘要进行中点击 `停止`，可中断摘要流程，不继续发送给 DS，主按钮恢复为 `发送` 或 `+`
- 发送成功后，待发送文件状态条自动清空
- 切换会话、新建会话、创建角色对话、创建群聊时，待发送文件状态条自动清空
- 第一档发送后，用户消息中显示问题正文，并以折叠附件块展示 txt 全文
- 第二档发送后，用户消息中显示问题正文，并以折叠附件块展示发送给 DS 的摘要内容
- 附件块默认折叠，点击后可展开查看全文或摘要
- 附件块中显示文件名、原始字数和发送模式（全文发送 / 摘要发送）
- 重新生成时沿用这条消息最终发送给 DS 的内容
- 编辑这条消息后，若内容被手动改写，则不再按附件块结构展示
- 导出会话时，txt 生成的最终用户消息会按普通消息一起导出
- 夜间模式下，`+` 主按钮、弹出菜单、状态条、附件折叠块样式正确
- 日间模式下，`+` 主按钮、弹出菜单、状态条、附件折叠块样式正确，文字和边框可读

- `DOMPurify.js` 加载失败时，聊天消息降级为纯文本显示，不崩溃

**二十四、请求超时控制**

- 单聊流式请求在连续 120 秒没有收到新 chunk 时自动中断，消息正文显示"请求超时，请检查网络后重试"
- 聊天请求超时后，消息底部显示红色状态文案"请求超时，请检查网络后重试"
- 超时中断后，发送按钮恢复正常状态（不再显示"停止"）
- 超时后的最终消息内容在 `renderChat()` 全量重渲染后仍然保留，不会变成空白消息
- 如果流式输出持续有新 chunk 到达，即使总耗时超过 120 秒，也不会仅因总时长被中断
- TXT 第二档摘要请求在连续 120 秒没有收到新 chunk 时自动中断，并停止后续发送
- 群聊中的路由判断、追问判断和角色回复生成在连续 120 秒没有收到新 chunk 时自动中断
- 正常请求（非超时）不受影响

- 群聊中编辑保存后，isSending 状态正确管理，不会卡死
- 群聊中编辑保存后，停止按钮可以正常中断生成
- 单聊中编辑用户消息后，inputMeta（token 统计）正确更新
- 编辑非最后一条用户消息时，正确截断后续所有消息

**三十七、群聊停止与中断**

- 群聊生成过程中点击"停止"按钮，立即中断所有 LLM 调用（路由、生成、追问）
- 中断后已生成的角色消息保留，未生成的不显示
- 中断的角色消息标记为"生成中断"
- 中断后 isSending 状态正确重置，可以继续发送新消息
- 路由判断被中断时，不会生成任何角色消息
- 角色生成被中断时，已完成的保留，当前中断的不保存
- 追问判断被中断时，已完成的保留

**三十八、群聊多轮互动**

- 用户发消息后，角色A回答 → 角色B回应 → 角色A再回应（多轮互动）
- 多轮互动最多 3 轮，不会无限循环
- 每轮最多一个角色回应
- 每个角色本轮最多说 2 次
- 任何一轮没人想说话时自动结束
- 多轮互动中角色能看到本轮其他角色已说的话（不重复）
- 多轮互动中停止按钮可以中断

**三十九、群聊导出**

- 群聊对话导出为 TXT 时，角色消息显示正确的角色名（不是"DeepSeek"）
- 用户消息显示为"我"
- 单聊消息导出不受影响

**三十九B、单角色聊天**

- 角色卡列表中每个角色有"单独聊天"按钮（聊天图标）
- 点击后创建新会话，tab 类型为 single-character，存储 characterId
- 点击后自动关闭角色卡管理面板和侧边栏
- tab 标题默认显示角色名（无自定义标题时）
- 单角色聊天不自动生成标题，tab 名称始终显示角色名（不会被动态生成的标题覆盖）
- 用户可手动重命名，重命名后显示自定义标题（覆盖角色名）
- tab 前显示 🎭 图标（与群聊 👥、单聊无图标区分）
- 聊天区顶部显示"正在与角色对话"+ 角色名彩色标签
- 发送消息时自动注入角色 system prompt（性格、背景、外貌、说话风格、口头禅）
- system prompt 通过 buildPayloadMessages 在 payload 最前面插入，不影响本地消息存储
- AI 回复以角色身份和风格回复（不是默认 DeepSeek 助手风格）
- AI 回复消息上方显示角色名标签（带角色颜色），左侧有彩色边框
- 流式生成过程中也显示角色名标签和彩色边框
- 流式完成后全量渲染（finalizeMessage）时角色名标签不丢失
- 支持深度思考模式（切换为 deepseek-reasoner）
- 支持编辑用户消息和重新生成
- 支持停止生成
- 支持版本历史切换
- 支持导出为 TXT（AI 回复显示角色名，不是"DeepSeek"）
- 删除角色时，如果该角色有单独聊天会话，弹出提示阻止删除（引用类型显示为"角色对话"）
- 删除角色后，已有对话中角色 system prompt 丢失（getCharacterById 返回 null），回退为普通对话
- 切换到其他 tab 再切回，消息正常显示（DOM 缓存正常）
- 单角色聊天的消息格式与单聊一致（用户蓝色气泡，AI 灰色气泡），不使用群聊的角色消息样式
- 单角色聊天支持 token 统计和上下文上限提醒（与单聊一致）
- 旧版数据（无 single-character 类型）不受影响

**四十一、数据兼容与边界**

- 旧版数据（tab 无 type 字段）默认作为单聊处理
- 群聊中旧数据（role 为 assistant 的消息）正确渲染为角色消息样式
- 删除角色前，如果该角色被对话或群聊引用，弹出提示要求先删除相关对话，不允许直接删除角色
- 删除未被引用的角色后，引用该角色的群聊不会崩溃（getCharacterById 返回 null 被 filter 过滤）
- 删除角色后，群聊历史消息中的角色名仍能正确显示（因为保存在消息数据中）
- localStorage 容量监控包含角色卡数据（dsCharacters）
- dsCharacters 数据损坏时自动重置为空数组
- 群聊名称为空时默认显示"群聊"
- 新建群聊面板每次打开时清空之前的选择
- 新建群聊面板每次打开时清空之前输入的群聊名称

**四十二、背景信息功能**

- 点击菜单中的"背景"可打开背景信息面板，且菜单自动关闭

**四十二B、背景信息注入 LLM**

- 单聊中设置了背景信息后，发送消息时 LLM 回复会体现背景信息（如称呼用户设定的角色名）
- 单聊中同时设置了角色名称和对话背景，两者都会注入到 system prompt
- 单角色对话中设置了背景信息，角色回复会融入背景设定
- 群聊中设置了背景信息，角色回复会融入背景设定（通过 groupContext 传递）
- 未设置背景信息时，对话行为与之前完全一致（不注入额外 system prompt）
- 背景信息作为 system prompt 注入，不受历史消息裁剪影响（始终存在）
- 背景信息中包含特殊字符（换行、引号等）时，不会导致 prompt 构建错误
